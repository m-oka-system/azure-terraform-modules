# 日本語技術文書スタイル

ドキュメント・説明・コメントを生成する際は、以下のルールに従うこと。
コード自体（変数名・関数名・HCL構文など）は英語のまま維持する。

## 文体ルール

### 基本方針

- 「です/ます」調で統一する。「だ/である」調は使わない
- 体言止めは見出し・箇条書きの項目名のみ許可する
- 一文は60文字以内を目安とし、長い場合は分割する
- 主語と述語を近づけ、修飾関係を明確にする

### 禁止事項

- 冗長な前置き（「それでは」「さて」「ちなみに」など）
- 二重否定（「〜ないわけではない」→「〜です」）
- 曖昧な表現（「〜と思います」→「〜です」「〜の可能性があります」）
- 過剰な敬語（「させていただきます」→「します」）
- 同じ文末の3回以上の連続（「〜です。〜です。〜です。」を避ける）

### 推奨表現

- 能動態を優先する（「設定されます」→「設定します」）
- 具体的な動詞を使う（「行う」→「実行する」「作成する」「設定する」）
- 技術用語は初出時に簡潔な説明を添える

## 構造ルール

### 見出し

- 見出しは体言止めにする（「設定方法」「前提条件」「動作確認」）
- 見出し階層は H2〜H4 の3段階までとする
- H1 はドキュメントタイトルのみに使用する

### 箇条書き

- 並列する項目は箇条書きにする（3項目以上の列挙は文中に書かない）
- 箇条書きの各項目は同じ品詞・文型で揃える
- ネストは2段階までとする

### テーブル

- 比較・一覧・対応関係にはテーブルを積極的に使う
- 列数は2〜5列を目安とする
- ヘッダー行は必須とし、簡潔な列名にする

### Mermaid ダイアグラム

- アーキテクチャ・フロー・シーケンスの説明には Mermaid を活用する
- ダイアグラム内のラベルは日本語で記述する
- ダイアグラムの直前に1行の説明文を添える
- 色やスタイリングは使用しない： `style`、`classDef`、`fill`、`stroke`、色の指定は避ける
- 使い分けの目安:
  - 処理の流れ → `flowchart` または `graph`
  - 時系列のやり取り → `sequenceDiagram`
  - 状態遷移 → `stateDiagram-v2`
  - クラス構成 → `classDiagram`

## 出力フォーマット

### 説明文の構成

1. 結論を先に述べる
2. 理由・背景を補足する
3. 具体例やコードを示す

### コードブロック

- 言語指定を必ず付ける（`hcl`、`bash` など）
- コード内のコメントは日本語で記述する
- 長いコードブロックにはセクション区切り（`####`）を使う

### エラー・警告の説明

- 原因 → 影響 → 対処法の順で記述する
- 対処法は具体的なコマンドまたはコード変更を示す
